많은 AI 데모는 "질문에 답하는 모델" 단계에서 멈춘다.  
내가 원한 것은 그보다 운영에 가까운 시스템이었다. **작업을 라우팅하고, 역할별 에이전트를 실행하고, 스스로 상태를 관측하며, 여러 채널로 알림을 전달하는 구조**.

이 프로젝트는 그 요구를 구현한 결과물이다.  
Discord 중심 인터페이스 위에 FastAPI 오케스트레이터, Redis 이벤트 백본, Ollama 기반 로컬 LLM 라우팅을 결합한 멀티 에이전트 플랫폼이다.

![Discord 멀티 에이전트 시스템 아키텍처 (Abstracted)](/images/ARCHITECTURE-ABSTRACT.png)

## 왜 만들었나

아래 문제가 반복됐다.

- 채팅/스크립트/대시보드로 흐름이 분산됨
- 에이전트 역할별 실행 계층이 통합되어 있지 않음
- 알림/백그라운드 자동화 신뢰성이 낮음
- 상시 실행 환경에서 운영 상태 추적이 어려움

그래서 한 플랫폼 안에 다음을 결합했다.

- 대화형 제어면(Discord)
- 에이전트 오케스트레이션(FastAPI + routing logic)
- 이벤트 백본(Redis pub/sub)
- 로컬 모델 실행(Ollama)
- 운영 가시성(Prometheus + Grafana)

## 시스템 요약

아키텍처는 의도적으로 모듈형이다.

- **Discord Bot**: 사용자 인터페이스(명령, 채널 기반 라우팅, 상태 UX)
- **Orchestrator (FastAPI)**: 에이전트 워크로드 실행 및 조정
- **Agent Layer**: 역할 특화 에이전트(orchestrator, stock, research, code, scheduler, system monitor)
- **Redis**: pub/sub 이벤트 + 경량 런타임 상태/이력
- **Observability**: Prometheus 메트릭 + Grafana 대시보드
- **Local LLM Runtime**: 에이전트별 모델 매핑 기반 Ollama 실행

이렇게 Discord는 단순 채팅 앱이 아니라 AI 워크플로 운영 콘솔이 된다.

## 실행 흐름 (Core Pattern)

대부분의 작업은 같은 패턴을 따른다.

1. 사용자 요청이 Discord 채널/슬래시 명령으로 들어온다.
2. Bot이 컨텍스트를 바탕으로 에이전트 타입을 매핑한다.
3. Orchestrator가 단일 혹은 멀티 에이전트 워크플로를 실행한다.
4. 에이전트가 결과를 내고, 필요 시 알림 이벤트를 발행한다.
5. Redis pub/sub가 이벤트를 구독자에게 전파한다.
6. 알림은 Discord와 선택 채널(Kakao)로 전달된다.
7. 메트릭/헬스 데이터는 지속적으로 수집된다.

이 패턴 덕분에 플랫폼 코어를 바꾸지 않고도 유스케이스를 확장할 수 있다.

## Local LLM-First Design

핵심 설계 결정 중 하나는 추론 계층을 local-first로 둔 것이다.

- Ollama가 로컬 인프라 안에서 모델을 서빙한다.
- Orchestrator는 에이전트 역할별로 작업을 라우팅한다(`MODEL_*` 매핑).
- 태스크 성격에 따라 서로 다른 모델을 쓸 수 있다(품질/속도 트레이드오프).
- 민감한 프롬프트/컨텍스트 데이터가 내부 환경에 머문다.
- 클라우드 전용 구성보다 비용/가용성이 예측 가능하다.

즉, 이건 단순한 "봇 자동화"가 아니다.  
모델 런타임을 직접 통제할 수 있는 **self-hosted AI operations stack**이다.

## 최근 개선한 점

### 1) 자유 텍스트 메모를 구조화 지식으로 전환

비정형 도메인 메모를 재사용 가능한 코드 레벨 스키마/포맷터로 전환했다.  
결과적으로 같은 데이터를 분석/요약/알림에 일관되게 재사용할 수 있게 됐다.

### 2) 멀티 채널 알림 표준화

다른 프로젝트에서 검증한 Kakao 알림 패턴을 이식했다.

- 개인 전송 / 워크스페이스 전송 모드
- 장문 자동 분할 전송
- 토큰 갱신 처리
- 추가 통합 방식(기존 Discord 흐름 유지)

이제 알림 전송은 단일 엔드포인트 하드코딩이 아닌 채널 독립 구조가 됐다.

### 3) 모델 레이어 업데이트 (Gemma)

코드 성향 작업에 맞춘 최신 Gemma 구성을 기본값에 반영했고, 이를 환경 기본값/모델 pull 스크립트/아키텍처 산출물까지 함께 업데이트했다.

## 이 프로젝트의 본질

이 프로젝트는 "주식 봇" 자체가 목적이 아니다.  
주식 모니터링은 여러 에이전트 경로 중 하나일 뿐이다.

실제 산출물은 다음을 갖춘 재사용 가능한 자동화 플랫폼이다.

- 에이전트 역할 분리와 라우팅
- 이벤트 기반 시스템 설계
- 로컬 모델 운영
- 신뢰성과 관측성 기본기
- 확장 가능한 알림/스케줄링 인프라

## 운영 교훈

- 운영 환경에서는 모델 지능보다 전달 신뢰성이 먼저다.
- 시간이 갈수록 비정형 텍스트보다 구조화 스키마가 강하다.
- 이벤트 백본(pub/sub)은 기능 간 결합도를 낮춘다.
- 단일 채널보다 멀티 채널 알림이 실전에서 안전하다.
- 관측성은 나중에 붙이는 기능이 아니라 초기 설계 항목이다.

## 다음 단계

- 알림 채널 테스트용 명시적 엔드포인트 추가
- 실패 알림 재시도/백오프/데드레터 처리 추가
- 에이전트 단위 텔레메트리 및 SLO 스타일 대시보드 강화
- 투자 외 도메인에도 재사용 가능한 워크플로 템플릿 패키징

한 줄 요약: 이번 구축의 우선순위는 도메인 전략보다 먼저, **전달 가능하고 확장 가능한 AI 자동화 인프라**를 확보하는 것이었다.
